The painful part

SUMO can finish a run while the experiment is still not comparable, reproducible, or safe to describe as a controller result.

What this skill changes

It makes the agent audit the experiment workflow before it writes code, trusts metrics, or upgrades a result into a paper-style claim.

Who should use it

Researchers building fixed-time, actuated, max-pressure, NEMA, data-informed, or MPC-style signal-control experiments in Eclipse SUMO.

Built for real SUMO failure modes

It catches the mistakes that usually appear after the paper draft is already written.

The skill is written around concrete failure patterns from SUMO documentation, community troubleshooting, public traffic-simulation code, and practical experiment experience.

  • Routes or demand differ across baselines.
  • TLS phase indexes do not match the intended movements.
  • Tripinfo, summary, or edgeData outputs are missing or overwritten.
  • A run stops early, teleports vehicles, or leaves unfinished demand.
  • Controller comparisons are not paired by seed, horizon, demand, and outputs.
  • The result is stronger than the evidence actually supports.

Current scope

The current release focuses on SUMO/TraCI traffic signal control experiments. It is not yet a general-purpose audit skill for every Eclipse SUMO use case.

Future skills may add audit support for routing and demand, emissions and energy, public transport, pedestrian and intermodal scenarios, AV/CAV and co-simulation workflows, calibration, safety analysis, and simulation-mode comparisons.

The core idea

It routes the work before it audits the work.

The agent should not use the same checklist for a new experiment, a failing run, a controller code change, and a claim-review request. This skill starts by identifying the situation, then loads only the subskill and reference modules needed for that path.

Ongoing project Project Control Screen -> target, state, deviation, next-step plan.
New experiment Socratic Intake -> Experiment Readiness Record -> SUMO Experiment Plan.
Code change TDD gate -> RED/GREEN/REFACTOR record -> verification review.
Failed or suspicious run Debugging helper -> symptom evidence -> fix, rerun, or claim demotion.
Results and claims Output gates -> metric definitions -> baseline pairing -> claim boundary.

Design approach

The skills combine official SUMO documentation, SUMO community troubleshooting patterns, public traffic-simulation code lessons, Socratic intake, confirmed experiment planning, TDD for SUMO/TraCI code changes, evidence-before-completion review gates, and privacy-safe field lesson capture.

Use scenarios

Start from the user's situation, then load only the references needed for that path.

  • Ongoing project: workflow router -> project control screen -> next-step plan.
  • New experiment: Socratic intake -> Experiment Readiness Record -> SUMO Experiment Plan.
  • Code change: TDD gate -> implementation -> verification and review gates.
  • Failed run: debugging helper -> closed-loop diagnosis -> fix or demotion.
  • Results or claims: output hard gates -> metrics/completion -> baseline design -> claim boundary.

Audit checks

  • Network, route, config, detector, and additional-file consistency.
  • TLS phase, movement-green, yellow, all-red, and NEMA evidence checks.
  • Paired controller comparisons across seeds, demand, horizons, and outputs.
  • Completion-aware reporting for stopped, unfinished, teleported, or invalid runs.
  • Metric definitions for travel time, delay, stops, throughput, queue, emissions, and fairness.
  • Claim boundaries for baselines, ablations, sensitivity runs, and debugging fixes.
  • Field lesson capture for user-discovered fixes that should become reusable skill guidance.
  • TDD records for SUMO/TraCI code changes before implementation claims.
  • Fresh verification evidence and review findings before completion claims.
  • Confirmed SUMO Experiment Plan after intake and before execution.

Limitations

This repository provides agent instructions, checklists, and audit procedures. It does not certify that a SUMO experiment is correct, publishable, or officially validated.
